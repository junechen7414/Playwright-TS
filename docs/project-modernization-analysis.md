## 🔍 現況分析

### 1. API Client 設計模式

#### 1.1 方法命名不一致

**問題描述**：
在 [`springboot-api-client.ts`](services/apis/springboot-api-client.ts:36-45) 中，列表查詢方法命名混用 `list` 和 `get` 前綴：

```typescript
// ❌ 不一致的命名
listAccounts(params?: {...}): Promise<ApiResult<PageResponse<...>>>  // 使用 list
listProducts(params?: {...}): Promise<ApiResult<PageResponse<...>>>  // 使用 list
listOrdersByAccount(accountId: number, params?: {...}): Promise<...> // 使用 list

getAccount(id: number): Promise<...>     // 使用 get（單一資源）
getProduct(id: number): Promise<...>     // 使用 get（單一資源）
getOrder(id: number): Promise<...>       // 使用 get（單一資源）
```

**影響**：
- 開發者需要記憶不同的命名模式
- 降低 API 的可預測性
- 不符合 RESTful 慣例（通常 `list` 用於列表，`get` 用於單一資源）

#### 1.2 回傳型別不一致

**問題描述**：
刪除操作的回傳型別不統一：

```typescript
// ❌ 不一致的回傳型別
deleteAccount(id: number): Promise<ApiResult<undefined | ApiError>>  // 回傳 undefined | ApiError
deleteProduct(id: number): Promise<ApiResult<void>>                  // 回傳 void
deleteOrder(id: number): Promise<ApiResult<void>>                    // 回傳 void
```

**影響範圍分析**：

經過實際檢查，`deleteAccount` 的回傳型別不一致主要影響：

1. **測試程式碼**（實際影響：無）
   - 檢查了所有使用 `deleteAccount` 的測試（2 處）
   - 都使用 `expectOk(response)` 處理回應
   - `expectOk` 可以處理 `void` 和 `undefined` 兩種型別
   - **結論**：修改回傳型別不會破壞現有測試

2. **型別一致性**（主要影響）
   - `deleteProduct` 和 `deleteOrder` 都回傳 `void`
   - 只有 `deleteAccount` 回傳 `undefined | ApiError`
   - 造成 API 設計不一致

3. **Biome Linter**（需驗證）
   - Biome 配置中啟用了 `recommended` 規則
   - 可能會對不一致的回傳型別發出警告
   - 但目前沒有配置特定的回傳型別檢查規則
   - **建議**：修改後執行 `pnpm biome check` 驗證

**實際測試使用情況**：
```typescript
// tests/api/springboot/account.spec.ts
test('應該能刪除帳戶', async ({ springbootApi, existingAccount }) => {
  const response = await springbootApi.deleteAccount(existingAccount.id);
  expectOk(response);  // ✅ expectOk 可以處理 void 和 undefined
  
  const getResponse = await springbootApi.getAccount(existingAccount.id);
  expectError(getResponse, 404);
});

test('當帳戶有關聯訂單時，無法刪除帳戶', async ({ ... }) => {
  const response = await springbootApi.deleteAccount(existingAccountWithOrders.id);
  const errorBody = expectError(response, 400);  // ✅ 錯誤處理不受影響
});
```

**結論**：
- 修改 `deleteAccount` 回傳型別為 `void` 是**安全的**
- 不會破壞現有測試程式碼
- 需要在修改後執行完整測試套件驗證
- 建議執行 `pnpm biome check` 確認 linter 無問題

#### 1.3 參數處理邏輯重複

**問題描述**：
更新操作中的參數處理邏輯重複且不一致：

```typescript
// ❌ 重複的參數處理邏輯
updateAccount(id: number, payload: Schemas['UpdateAccountRequest']): Promise<ApiResult<void>> {
  const params: Record<string, string | number | boolean> = {};
  if (payload.name) params.name = payload.name;
  if (payload.status) params.status = payload.status;
  return this.requester.put<void>(`${this.endpoints.account}/${id}`, { data: params });
}

updateProduct(id: number, payload: Schemas['UpdateProductRequest']): Promise<ApiResult<void>> {
  const params: Record<string, string | number | boolean> = {
    price: payload.price,
    saleStatus: payload.saleStatus,
    available: payload.available,
  };
  if (payload.name) params.name = payload.name;
  return this.requester.put<void>(`${this.endpoints.product}/${id}`, { data: params });
}
```

**影響**：
- 程式碼重複，違反 DRY 原則
- 維護成本高（修改邏輯需要多處更新）
- 容易產生不一致的行為

### 2. 型別安全與 Schema 定義

#### 2.1 PageResponse 定義位置問題

**問題描述**：
[`PageResponse`](services/schema/api-types.ts:347-358) 介面定義在自動生成的檔案中：

```typescript
/**
 * This file was auto-generated by openapi-typescript.
 * Do not make direct changes to the file.
 */

// ... 自動生成的型別 ...

/**
 * 分頁回應的通用型別
 */
export interface PageResponse<T> {
  content: T[];
  page: number;
  size: number;
  totalElements: number;
  totalPages: number;
}
```

**Git 歷史分析**（實際證據）：

經過 git log 調查，確認 `PageResponse` 的演變歷史：

1. **首次加入**：[`d819a81`](https://github.com/commit/d819a81) - `feat(api): add pagination support for listOrdersByAccount`
   - 日期：2026-06-09 15:34:22
   - 作者手動加入 `PageResponse` 型別到自動生成的檔案中
   - 原始結構包含完整的 Spring Data Page 物件（pageable, sort 等）

2. **修正定義**：[`e6f8246`](https://github.com/commit/e6f8246) - `fix(api): correct PageResponse type definition`
   - 日期：2026-06-09 16:31:58
   - 簡化為符合後端實際回應的結構（移除 pageable, sort 等複雜欄位）
   - 加入詳細的中文 JSDoc 註解

3. **自動生成檔案**：[`25158fe`](https://github.com/commit/25158fe) - `執行openapi-typescript指令產出ts type`
   - 最初的自動生成檔案**不包含** `PageResponse`
   - 證明 `PageResponse` 是後來手動加入的

**結論**：
- ✅ **確認**：`PageResponse` 是手動維護的型別
- ❌ **問題**：放在自動生成檔案中，違反「不要修改自動生成檔案」的原則
- ⚠️ **風險**：重新執行 `openapi-typescript` 時會遺失此型別定義

**影響**：
- 重新生成 API 型別時會遺失 `PageResponse` 定義
- 手動維護的型別與自動生成的型別混在一起
- 違反「不要修改自動生成檔案」的原則
- 需要記得在重新生成後手動加回

**建議**：
- 將 `PageResponse` 移到獨立的 `common-types.ts` 檔案
- 確保重新生成 API 型別時不會遺失
- 符合「分離關注點」的原則

## 🎯 現代化設計原則

### 原則 1: 一致性優先

**理由**：降低認知負擔、提升程式碼可預測性、減少錯誤使用

**應用**：
- API 方法命名統一
- 回傳型別統一
- 定位器策略統一
- 測試案例命名統一

### 原則 2: 型別安全

**理由**：在編譯時發現錯誤、提供更好的 IDE 支援、減少執行時錯誤

**應用**：
- 使用 `const enum` 減少 bundle 大小
- 環境變數使用型別定義和驗證
- 分離手動維護的型別和自動生成的型別

### 原則 3: 可維護性

**理由**：降低維護成本、提升程式碼品質、便於重構和擴展

**應用**：
- 使用路徑別名簡化 import
- 提取共用邏輯避免重複
- 加入 JSDoc 註解提升可讀性

### 原則 4: 最佳實踐

**理由**：符合業界標準、提升團隊協作效率、便於新成員上手

**應用**：
- 遵循 Playwright 定位器最佳實踐
- 遵循 TypeScript 最佳實踐
- 遵循測試命名規範

### 原則 5: 文件同步

**理由**：確保文件可信度、降低學習成本、提升開發效率

**應用**：
- 文件與程式碼保持同步
- 範例程式碼可直接使用
- 提供完整的使用指南

#### 1.3 提取共用參數處理邏輯

**重構方案**：

```typescript
export class SpringbootApiClient {
  /**
   * 過濾掉 undefined 的屬性
   * @param obj - 要過濾的物件
   * @returns 過濾後的物件
   */
  private filterUndefined<T extends Record<string, unknown>>(obj: T): Partial<T> {
    return Object.fromEntries(
      Object.entries(obj).filter(([_, value]) => value !== undefined)
    ) as Partial<T>;
  }

  updateAccount(id: number, payload: Schemas['UpdateAccountRequest']): Promise<ApiResult<void>> {
    const params = this.filterUndefined(payload);
    return this.requester.put<void>(`${this.endpoints.account}/${id}`, { data: params });
  }
```
