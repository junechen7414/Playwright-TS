## 步驟 5：進階：Custom Fixture 與 整合 (mergeTests)

Playwright 的 Fixture 系統可以讓我們封裝重複的初始化邏輯，使測試案例更簡潔且易於維護。

### 核心優勢
1.  **封裝與可重用性 (Encapsulation)**：Fixture 將「準備工作」與「測試內容」徹底分離。例如定義一個 `adminPage` fixture，即可包含所有登入管理後台的步驟。
2.  **依賴注入 (Dependency Injection)**：只需在測試函數的參數中「宣告」需要的 fixture，Playwright 就會自動執行相關的初始化邏輯。
3.  **按需執行 (Lazy Loading) - 最關鍵**：`beforeEach` 無論測試內容為何都會執行；而 **Fixture 只有在測試函數有用到它時，才會被執行**。
    *   **彈性定義**：相較於傳統 `beforeEach`/`afterEach`，Fixture 透過 `await use()` 作為分隔線，在同一個定義中完成測試前置（Setup）與測試後清理（Teardown）。

> 參考教學：[Playwright Fixtures 核心概念 (YouTube)](https://youtu.be/wXHiq9H3MB0)

---

### 1. 建立 Custom Fixture (`test.extend`)

使用 `@playwright/test` 的 `test.extend<T>()` 來擴充。
可以針對不同需求建立獨立的 Fixture 檔案：
- **Page Objects (`PageObjects.fixture.ts`)**：封裝所有 UI 頁面物件。
  ```typescript
  export const pageObjectTest = baseTest.extend<PageObject>({
    loginPage: async ({ page }, use) => {
      await use(new LoginPage(page));
    },
  });
  ```
- **API Clients (`ApiObjects.fixture.ts`)**：封裝 API 請求與**自動清理 (Teardown)** 邏輯。
  ```typescript
  bookingApiWithDataDeletedAfterward: async ({ bookingApi }, use) => {
    const createResponse = await bookingApi.createBooking(newBookingPayload);
    const bookingId = (await createResponse.json()).bookingid;
    await use({ bookingApi, bookingId });
    // 測試結束後自動刪除資料 (Teardown)
    await bookingApi.deleteBooking(bookingId);
  },
  ```
- **測試資料 (`TestData.fixture.ts`)**：根據環境變數 (`process.env.ENV`) 提供對應的測試資料。

### 2. 鏈結與整合 Fixture (`mergeTests`)
當專案中有多個 Fixture 檔案時，可以使用 `mergeTests` 將它們整合在一個 `test` 物件中，方便測試案引用。
- **範例 (`chainFixtures.ts`)**：
  ```typescript
  import { mergeTests } from '@playwright/test';
  import { pageObjectTest } from './PageObjects.fixture';
  import { sauceTest } from './TestData.fixture';

  // 整合 UI 與 Saucedemo 測試資料 Fixtures
  export const test = mergeTests(pageObjectTest, sauceTest);

  // 同時匯出 expect，讓 spec 檔案只需一行 import 即可搞定
  export { expect } from '@playwright/test';
  ```

### 3. 運作機制說明
- **`await use(object)`**：
    1.  將物件提供給依賴它的測試使用。
    2.  **暫停**執行該 Fixture。
    3.  等待**測試案例執行完畢**後，繼續執行 `use()` 之後的代碼（即 Teardown 邏輯）。
- **參數語法**：測試函數接收一個物件參數 `{ loginPage, ... }`，裡面包含所有要使用的 fixtures。

### 4. 在測試中使用整合後的 Fixture
撰寫測試案時，只需從 `chainFixtures.ts` 匯入 `test`，即可同時獲得所有已定義的 Fixtures：
```typescript
import { test, expect } from '../fixtures/chainFixtures';

test('範例測試', async ({ loginPage, standardUserData }) => {
  await loginPage.login(standardUserData.username, standardUserData.password);
  // ...
});
```

## 步驟 6：定位元素最佳實踐 (Locators)

Playwright 官方強烈建議使用 **User-Facing Locators**，這類定位器模擬使用者的互動方式（如「點擊按鈕」、「尋找特定文字」），而非依賴不穩定的 CSS Class 或 XPath。

### 1. 優先使用的 Locators (推薦順序)
- **`page.getByRole(role, options)` (首選)**：
  透過 ARIA role 定位元素，這是最穩健且符合無障礙設計 (A11y) 的方式。
  ```typescript
  // 點擊名稱為 "登入" 的按鈕
  await page.getByRole('button', { name: '登入' }).click();
  // 勾選 "同意條款" 的核取方塊
  await page.getByRole('checkbox', { name: '同意條款' }).check();
  ```
- **`page.getByText(text)`**：
  透過顯示的文字內容定位。
  ```typescript
  await page.getByText('歡迎回來').isVisible();
  ```
- **`page.getByLabel(label)`**：
  適用於表單輸入欄位 (Input)，透過關聯的 Label 文字定位。
  ```typescript
  await page.getByLabel('使用者名稱').fill('admin');
  ```
- **`page.getByPlaceholder(text)`**：
  透過 placeholder 屬性定位（若無 Label 時使用）。

### 2. 搭配 Chrome DevTools 查看 Accessibility
要精準使用 `getByRole`，必須知道元素的 **Role (角色)** 與 **Accessible Name (名稱)**。

1.  **開啟 Chrome DevTools**：
    在網頁上按 `F12` 或右鍵選「檢查 (Inspect)」。
2.  **使用 Accessibility 面板**：
    - 在 **Elements** 面板右側的子分頁中，切換到 **Accessibility** 標籤（若沒看到，點擊 `>>` 展開選單尋找）。
    - 選取 DOM 元素後，查看 **Computed Properties** 區塊：
        - **Role**：例如 `button`, `link`, `heading`。
        - **Name**：該元素的識別名稱 (來自 text content, aria-label, alt 等)。
    - *這就是 `getByRole('Role', { name: 'Name' })` 的參數來源。*

## 步驟 7：Page Object Model (POM) 設計原則：Business-Layer 概念

本專案的 POM 設計核心參考了 **Business-Layer Page-Object Pattern**。其目標是讓測試腳本（Test Specs）擁有清晰的結構，專注於「做了什麼」而非「如何操作」。

### 核心理念
> *「我們不在測試中捕捉選擇器 (Selectors)，也不直接操作清單，而是調用具有語意化名稱的函數。這讓不具備深厚技術背景的人也能理解測試在執行什麼。」*

### 實作守則
1.  **封裝低階操作 (Encapsulate Low-level Actions)**：
    Page Object 內部的 `Locator` 與 `page.click()` 等實作細節不應暴露給測試層。
2.  **商業邏輯導向 (Business Logic Oriented)**：
    方法命名應反映使用者的行為，而非 UI 元件的操作。例如使用 `fillCheckoutInformation()` 而不是 `fillFirstName()`, `fillLastName()`。
3.  **隱藏斷言細節 (Hiding Assertion Details)**：
    將常用的驗證邏輯封裝進 Page Object 中（如 `verifyOrderCompletion()`），使測試案例保持簡潔。

### 範例：`CheckoutPage.ts` 的實作
觀察以下 `CheckoutPage` 的方法定義，即使不看程式碼實作，也能一眼看出測試流程：

```typescript
// 測試案例中的寫法
await checkoutPage.fillCheckoutInformation('John', 'Doe', '12345');
await checkoutPage.continueCheckout();
await checkoutPage.finishCheckout();
await checkoutPage.verifyOrderCompletion();
```

**這樣設計的好處：**
-   **可讀性 (Readability)**：測試腳本就像在閱讀商業流程。
-   **可維護性 (Maintainability)**：若 UI 結構改變（例如 Input ID 變更），只需修改 Page Object 中的一處，所有測試案例皆能同步更新。
-   **降低門檻 (Lower Barrier to Entry)**：非開發人員（如 QA 或 PM）也能輕鬆理解測試邏輯。

## 步驟 8：常用 API 技巧與知識點

在實戰開發中，利用 Playwright 的進階 API 可以大幅提升測試的穩定性與效率。

### 1. 全域登入狀態分享 (`storageState`)
為了避免每個測試案都重複執行登入流程，可以先儲存 Cookie 與資訊提供給其他案例使用。
- **運作流程**：
    1.  建立一個 **Setup 測試** 執行登入。
    2.  呼叫 `await page.context().storageState({ path: 'auth.json' })`。
    3.  在 `playwright.config.ts` 中設定 `dependencies` 確保 Setup 先跑，並在 `use` 中設定 `storageState: 'auth.json'`。
- **優點**：大幅縮減測試執行時間，達成一次登入、全域使用。

### 2. 自動處理隨機彈窗 (`addLocatorHandler`)
處理隨機跳出的元素（如：Cookie 確認框、訂閱電子報、第三方授權同意框）。
```typescript
// 當發現 cookie 彈窗出現時，自動點擊關閉
await page.addLocatorHandler(
  page.getByRole('button', { name: '同意' }),
  async () => {
    await page.getByRole('button', { name: '關閉' }).click();
  }
);
```

### 3. 頁面跳轉等待優化 (`goto`)
`page.goto(url, { waitUntil: '...' })` 的預設參數是 `'load'` (等待所有資源載入)。
- **推薦參數**：
    - `'domcontentloaded'`：HTML 下載完且 DOM 樹建立完成即可（不需等待圖片、CSS 載入完畢）。
    - `'commit'`：瀏覽器接收到回應並開始跳轉即視為完成（最快）。
- **注意**：若頁面高度依賴動態載入，建議仍使用預設值或搭配 `waitForSelector`。

### 4. 斷言技巧 (`expect`)
- **軟斷言 (`expect.soft`)**：
    - 當第一個斷言失敗時**不會中止測試**，會繼續執行後續步驟。
    - 適用於單一測試案例中有多個檢查點，且希望一次看完所有失敗原因。
- **API 狀態驗證 (`expect.toBeOK`)**：
    - 專門用來驗證 API Response 的 Status Code 是否在 `200-299` 之間。
    ```typescript
    const response = await request.get('/api/data');
    await expect(response).toBeOK();
    ```

### 5. 網路攔截與 API Mocking (`page.route`)
測試前端介面時，常遇到「如何測試伺服器錯誤？」或「如何測試特定資料狀態？」的問題。使用 `page.route` 可以攔截網路請求並偽造回應 (Mock)。

- **適用情境**：
    - **模擬 API 錯誤**：驗證當後端回傳 401 (未授權) 或 500 (伺服器錯誤) 時，UI 是否有顯示對應的錯誤提示。
    - **固定測試資料**：不管後端資料庫為何，強制 API 回傳特定的 JSON，確保 UI 測試的一致性。
    - **模擬網路中斷**：使用 `route.abort()` 測試離線狀態的處理。

```typescript
// 攔截所有發往 '/api/user' 的請求，並強制回傳 500 錯誤
await page.route('**/api/user', async route => {
  await route.fulfill({
    status: 500,
    contentType: 'application/json',
    body: JSON.stringify({ message: 'Internal Server Error' }),
  });
});
```

### 補充：如果不使用 Fixture，程式碼會長怎樣？
在大型專案中，若不使用 Fixture，你可能需要在每個測試檔案中手動管理 Page Objects 的生命週期，導致程式碼變得冗長且難以維護：

```typescript
let loginPage: LoginPage;
let inventoryPage: InventoryPage;

test.describe('Shopping Flow Tests', () => {
  test.beforeEach(async ({ page }) => {
    loginPage = new LoginPage(page);
    inventoryPage = new InventoryPage(page);
    
    // 執行共用的前置操作
    await loginPage.goto();
    await loginPage.login(USER, PASS);
  });

  test('測試案例 1', async () => {
    // 必須依賴外部變數 loginPage
    await inventoryPage.addItemToCart();
  });
});
```
**缺點：**
- **變數污染**：需要在 `describe` 作用域宣告 `let` 變數。
- **強制執行**：即使某個測試案例不需要 `inventoryPage`，`beforeEach` 仍會初始化它。
- **難以分享**：這段初始化邏輯很難在不同的測試檔案間優雅地共享。

## 步驟 9：核心配置詳解 (`playwright.config.ts`)

`playwright.config.ts` 是控制測試行為的中樞，本專案針對「多環境切換」、「CI/CD 整合」與「執行效率」做了深度客製化。

### 1. 專案依賴與執行順序 (Projects & Dependencies)
Playwright 支援將測試拆分為不同的 `Project`，並設定彼此的依賴關係。這在本專案中用於處理 **Global Setup (登入驗證)**。

- **Setup 模式**：
  定義一個名為 `ui-setup` 的 Project 專門執行登入腳本 (`*.setup.ts`) 並儲存狀態 (`storageState`)。
- **依賴關係**：
  主測試 Project (`ui-saucedemo-chromium`) 設定 `dependencies: ['ui-setup']`。
  **效果**：Playwright 會自動先跑 Setup，成功後才跑主測試，並自動注入登入狀態。

### 2. CI/CD 與 本機環境的策略差異
設定檔透過 `process.env.CI` 判斷當前是否在 CI 環境執行，並動態調整參數：

| 設定項目 | CI 環境 (GitHub Actions) | 本機開發 (Local) | 目的 |
| :--- | :--- | :--- | :--- |
| **Retries** | 2 次 | 0 次 | CI 重試以排除網路波動；本機除錯不需重試 |
| **Workers** | 2 | 3 (或更多) | CI 資源有限需降速求穩；本機追求速度 |
| **Reporter** | List + HTML | List | CI 需產出報表供檢視；本機直接看 Terminal |
| **Video/Trace**| Retain on Failure | Off / On | 節省 CI 儲存空間，僅失敗時保留證據 |

### 3. 失敗證據保留 (Artifacts)
為了節省資源並精準除錯，我們採用「只在失敗時保留」的策略：
```typescript
screenshot: 'on-first-failure', // 只有失敗時截圖
video: {
    mode: 'retain-on-failure',  // 只有失敗時保留影片
},
```

## 步驟 10：GitHub Actions CI/CD 設定與本地驗證

本專案配置了 GitHub Actions (.github/workflows/playwright.yml) 來實現自動化測試。此流程特別針對效能優化，並使用官方 Docker 映像檔來確保環境一致性。

### 0. 本地開發與 GHCR 認證 (Local Auth)
在本地執行 `podman compose` 前，必須先確保具備存取 GitHub Container Registry (ghcr.io) 的權限。

* **認證方式**：無法使用 GitHub 登入密碼，必須使用 **Personal Access Token (PAT)**。
* **PAT 最小權限原則**：
    * 進入 GitHub Settings -> Developer settings -> PAT (Classic)。
    * 僅需勾選 `read:packages` (讀取) 或 `write:packages` (推送)。
* **登入指令**：
    ```bash
    # 輸入後，密碼欄位貼上 PAT (ghp_...)
    podman login ghcr.io
    ```

### 1. 核心架構：官方容器化執行 (Containerized Execution)
我們不在 CI 的虛擬機 (VM) 上花時間安裝瀏覽器，而是直接在微軟提供的 Playwright Docker 容器中執行測試。
- **Image**: `mcr.microsoft.com/playwright:v1.57.0-noble`
- **優勢**:
  - **預先安裝瀏覽器**：省去 `npx playwright install` 的下載時間。
  - **環境一致性**：確保 CI 執行環境與本地依賴版本完全匹配。
  - **系統依賴**：容器內已包含執行瀏覽器所需的所有 Linux 函式庫。

### 2. Workflow 關鍵步驟解析

#### 觸發條件 (Triggers)
設定為當 Pull Request 合併至 `main` 或 `master` 分支時自動觸發，同時保留手動觸發 (`workflow_dispatch`) 選項。 

#### 詳細步驟說明
1.  **容器初始化**：
    直接指定 `container`，讓後續所有 `run` 指令都在 Docker 容器內執行。

2.  **依賴快取 (Caching Strategy)**：
    針對 `pnpm store` 進行快取，而非傳統的 `node_modules` 快取。這能顯著減少 I/O 讀寫並加速依賴安裝。
3.  **安裝與執行**：
    - 使用 `--frozen-lockfile` 確保安裝版本鎖定。
    - 設定 `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD: 1`，告知 Playwright 直接使用容器內的瀏覽器（節省頻寬與時間）。
4.  **報告保存 (Artifacts)**：
    使用 `if: always()` 確保即使測試失敗，也能上傳測試報告以供除錯。
